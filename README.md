## Overview

This repository contains a working prototype simulator for evaluating how measurement uncertainty affects clinical AI systems built from electronic medical record (EMR) data.

Clinical AI pipelines often treat recorded laboratory values, vital signs, and diagnostic labels as if they directly represent biological truth. In practice, these values can be affected by specimen handling, laboratory artifacts, device calibration, transient physiologic variation, and threshold-based definitions.

The DBbun Measurement-Aware FHIR Companion simulates these effects using synthetic clinical data and FHIR R4-annotated outputs. It helps researchers, clinical AI developers, health systems, and standards developers test whether AI labels, phenotype definitions, or risk classifications remain stable under plausible measurement uncertainty.

## Prototype Status

This is a **working prototype** generated by the DBbun platform from a source clinical AI paper on EMR measurement uncertainty.

The current bundle includes:

- 5 simulation scenarios
- 2,000 synthetic patients
- 20 serial draws per patient
- 124,000 total simulation rows
- 23 documented simulation parameters
- 8 generated figures
- 5 standalone FHIR Bundle JSON examples
- Reproducible Python code and CSV/JSON outputs

## Core Scenarios

| Scenario | Focus | Main issue demonstrated |
|---|---|---|
| 1 | False critical potassium | Hemolyzed specimens can create false critical potassium labels. |
| 2 | Borderline HbA1c | Patients near the diabetes threshold can flip labels across serial draws. |
| 3 | Borderline systolic blood pressure | White-coat effect and device noise can create unstable hypertension labels. |
| 4 | Cross-site calibration drift | The same synthetic patients can show very different apparent prevalence across sites. |
| 5 | Threshold sensitivity sweep | Label noise peaks near clinical diagnostic boundaries. |

## Key Findings from the Prototype

| Scenario | Key uncertainty source | Label error rate | Notable finding |
|---|---|---|---|
| False critical potassium | Hemolysis in 7.6% of draws | 3.7% | False critical values occurred despite normal true potassium. |
| Borderline HbA1c | Inter-assay variation and calibration bias | 28.3% | 82.8% of borderline patients flipped diabetes labels across 20 draws. |
| Borderline SBP | White-coat effect and device noise | 30.2% | White-coat-positive patients were systematically over-labeled as hypertensive. |
| Cross-site calibration drift | Site calibration bias | 35.3% | Apparent diabetes prevalence ranged from 93% to 0% across sites using the same patients. |
| Threshold sensitivity sweep | Threshold perturbation | Up to 47.9% | Label noise peaked near current guideline boundaries. |

## HL7 and Terminology Standards

The prototype uses **FHIR R4** as the primary interoperability structure.

FHIR resources represented in the bundle include:

- `Patient`
- `Observation`
- `Condition`
- `Specimen`
- `Device`
- `Provenance`
- `Bundle`

Terminology standards include:

- **LOINC** for laboratory and vital sign observations
- **SNOMED CT** for clinical conditions and specimen concepts
- **UCUM** for measurement units

The FHIR examples show how measurement context can become computable, not just descriptive. For example, a potassium observation can reference a hemolyzed specimen, a device/analyzer, an interpretation code, and a measurement-reliability extension.

## Quick Start

Install the required Python packages:

```bash
pip install numpy matplotlib
```

Run the simulator:

```bash
python Simulator.py
```

Specify a custom output directory:

```bash
python Simulator.py --output ./my_results
```

The simulator is self-contained. It does not require an API key, network connection, EHR system, or real patient data.

## Outputs

The simulator produces:

- `sim_outputs/simulation_outputs.csv` — row-level synthetic observations and labels
- `sim_outputs/scenario_summary.csv` — aggregate results by scenario
- `sim_outputs/parameters_used.csv` — simulation parameters, units, and provenance
- `sim_outputs/summary.json` — key metrics and run metadata
- `sim_outputs/*.png` — generated figures
- `fhir_bundles/*.json` — FHIR Bundle JSON examples for all five scenarios

## Why This Matters

Analytic reproducibility is not enough for clinical AI. A model can reproduce the same outputs from the same dataset while still encoding measurement artifacts embedded in the data.

This prototype demonstrates how measurement uncertainty can be represented, simulated, and evaluated before deployment. It supports a more transparent approach to clinical AI validation by making measurement context, label confidence, and provenance explicit.
